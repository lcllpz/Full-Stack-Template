# 密码加密方案总结

本文结合本仓库 `apps/back` 的注册与登录实现，说明密码存储方式、bcryptjs 选型理由，以及与 Session `hash` 的区别。

---

## 1. 注册时密码如何加密？

### 1.1 结论

本项目注册时使用的是 **bcryptjs 加盐哈希**，**不是 MD5**。

全仓库未使用 MD5 处理密码。

### 1.2 注册流程

`AuthService.register()` 调用 `UserService.create()` 创建用户。密码在 `UserService.create()` 中处理：

```typescript
// apps/back/src/user/user.service.ts
if (createUserDto.password) {
  const salt = await bcrypt.genSalt(10);
  password = await bcrypt.hash(createUserDto.password, salt);
}
```

步骤：

1. `bcrypt.genSalt(10)` — 生成盐，cost factor = 10
2. `bcrypt.hash(明文, salt)` — 对明文密码做加盐哈希
3. 将哈希结果写入数据库，**不存明文**

### 1.3 登录校验

登录时在 `AuthService.login()` 中用 `bcrypt.compare()` 比对：

```typescript
// apps/back/src/auth/auth.service.ts
const isPasswordValid = await bcrypt.compare(loginDto.password, user.password);
```

`compare` 会从存储的哈希字符串中解析盐，再对输入密码做同样运算后比对，无需单独查盐字段。

---

## 2. 为什么不用 MD5？

| 对比项     | MD5                          | bcryptjs                     |
| ---------- | ---------------------------- | ---------------------------- |
| 设计目标   | 快速摘要，适合校验完整性     | 专为密码存储设计             |
| 加盐       | 需手动实现                   | 内置，每次哈希自动生成随机盐 |
| 计算速度   | 极快，GPU 可每秒穷举数十亿次 | 故意慢，可通过 cost 调节     |
| 彩虹表攻击 | 易被预计算字典攻击           | 每个密码盐不同，难以预计算   |
| 行业实践   | 已淘汰，不应用于密码         | 广泛使用的密码哈希方案       |

即使「MD5 + 固定盐」也不安全：盐一旦泄露，全库密码面临风险，且 MD5 本身仍然太快。

---

## 3. bcryptjs 的核心优势

### 3.1 内置加盐，每个密码结果不同

bcrypt 自动为每次哈希生成随机盐，并将盐编码进结果字符串（形如 `$2a$10$...`）。

- 两个用户密码相同，数据库存储值也不同
- 彩虹表、预计算字典攻击基本失效
- 无需自行设计「盐存哪、怎么拼」

### 3.2 故意「慢」，抗暴力破解

bcrypt 基于 Blowfish，通过 **cost factor**（本项目为 `10`）控制计算轮数：

- cost 每 +1，耗时大约翻倍
- 正常登录时 `compare` 略慢可接受
- 攻击者批量穷举时成本急剧上升

MD5 / SHA 追求**快**，不适合存储密码。

### 3.3 自适应，可随硬件升级调参

机器算力提升后，可将 `genSalt(10)` 调整为 `12`、`14` 等，提高新注册用户的哈希强度，无需更换整套方案。

### 3.4 生态成熟，API 不易误用

- 长期用于密码存储，实践验证充分
- `bcryptjs` 为纯 JS 实现，跨平台、无原生编译依赖
- `hash` / `compare` 语义清晰，降低实现错误风险

---

## 4. 与其他密码哈希方案对比

| 方案              | 适合存密码？ | 说明                                  |
| ----------------- | ------------ | ------------------------------------- |
| MD5 / SHA-256     | 否           | 太快、无内置盐、易被彩虹表攻击        |
| 自行 MD5 + 固定盐 | 否           | 盐泄露后全库危险；仍过快              |
| bcrypt / scrypt   | 是           | 慢哈希 + 随机盐，可调 cost            |
| Argon2id          | 是           | 内存硬、抗 GPU 更强，适合更高安全要求 |

对大多数业务，**bcrypt 已足够**；若追求更高标准，可考虑 Argon2id。

---

## 5. 易混淆概念：密码哈希 vs Session hash

本项目中有两类「hash」，用途完全不同，不可混用。

| 类型             | 生成方式                          | 用途                                 | 相关代码                                |
| ---------------- | --------------------------------- | ------------------------------------ | --------------------------------------- |
| **用户密码哈希** | `bcrypt.hash()`                   | 注册存储、登录校验                   | `user.service.ts`、`auth.service.ts`    |
| **会话 hash**    | `randomBytes(32).toString('hex')` | 绑定 refresh token，刷新时校验与轮换 | `session.service.ts`、`auth.service.ts` |

Session `hash` 用于 JWT + Session 认证体系中的令牌刷新与防重放，**不是**用户密码的加密方式。详见 [session-认证方案总结.md](./session-认证方案总结.md)。

## 7. 小结

1. 注册密码使用 **bcryptjs + 盐（cost = 10）**，不存明文，不使用 MD5。
2. bcryptjs 的优势：**慢哈希 + 自动加盐 + 可调 cost + 成熟 API**，针对密码场景设计。
3. Session 中的 `hash` 是会话令牌校验，与用户密码加密无关。
4. authentication： http://www.passportjs.org/concepts/authentication/downloads/html/
