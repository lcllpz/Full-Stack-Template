# 邮箱 / 手机验证码为啥要走队列

本文结合本仓库 `apps/back` 的验证码发送实现，说明为什么用 **BullMQ 异步队列**，而不是在 HTTP 接口里直接调 SMTP / 短信 API。

> 队列通用知识见 [`docs/techniques/queues.md`](../../techniques/queues.md)。  
> 整体认证增强方案见 [`docs/plan/auth-enhancement-plan.md`](../../plan/auth-enhancement-plan.md)。

---

## 1. 一句话结论

验证码发送是 **慢、易失败、可重试** 的副作用，不适合堵在登录/注册接口里。  
做法是：**验证码先写入 Redis，发送动作入队**，由 Worker 异步消费；HTTP 尽快返回「已发送」。

---

## 2. 同步发送的问题

若在 `AuthService` 里直接 `await mailService.send(...)` / `await smsService.send(...)`：

| 问题         | 表现                                                                |
| ------------ | ------------------------------------------------------------------- |
| 响应慢       | SMTP / 短信供应商常要几百毫秒到数秒，用户一直转圈                   |
| 失败难恢复   | 临时网络抖动、供应商限流 → 接口直接 5xx，只能让用户再点「重新发送」 |
| 占用线程     | 发送卡住会占住 Nest 处理请求的能力，高峰时拖垮其它接口              |
| 易打爆第三方 | 突发大量发码请求会直接打到 SMTP / 短信商，触发限流或封禁            |
| 职责混杂     | Auth 既要校验身份，又要关心邮件模板、重试策略、供应商细节           |

---

## 3. 走队列带来的好处

| 能力     | 说明                                                                  |
| -------- | --------------------------------------------------------------------- |
| 快速响应 | 接口只做：校验图形验证码 → `issueCode` 落 Redis → `enqueue`，即可返回 |
| 自动重试 | Job 配置 `attempts: 3` + 指数退避，临时故障可自动再试                 |
| 削峰     | Worker 按并发消费，避免瞬时打满供应商额度                             |
| 解耦     | 生成/校验验证码 与 真正发信/发短信 分离                               |
| 可扩展   | 以后可拆独立 worker 进程，API 与发送互不影响                          |

---

## 4. 本项目里的分工

```
发码接口
  → CaptchaService（图形验证码前置）
  → VerificationService.issueCode（生成 + 存 Redis + 频率/日限）
  → NotificationService.enqueue*（投递 BullMQ Job）
  → MailProcessor / SmsProcessor（异步真正发送）
       → MailService / SmsService
```

| 模块                  | 职责                                           |
| --------------------- | ---------------------------------------------- |
| `VerificationService` | 只管码：生成、哈希存储、TTL、防重放、频率限制  |
| `NotificationService` | 只管入队：邮件队列 / 短信队列                  |
| `MailProcessor`       | 消费 `mail` 队列，调 `MailService`             |
| `SmsProcessor`        | 消费 `sms` 队列，调 `SmsService`               |
| `AuthService`         | 编排业务：发码场景、登录/绑定/换绑等，不直接发 |

关键代码：

```typescript
// apps/back/src/queue/notification.service.ts
private readonly defaultJobOptions = {
  attempts: 3,
  backoff: { type: 'exponential' as const, delay: 2000 },
  removeOnComplete: 50,
  removeOnFail: 100,
};

async enqueueEmailCode(to: string, code: string, purpose = '验证码') {
  await this.mailQueue.add(JOB_SEND_EMAIL_CODE, { to, code, purpose }, this.defaultJobOptions);
}
```

`AuthService` 里发码统一是：先 `issueCode`，再 `enqueueEmailCode` / `enqueueSmsCode`（注册、验证码登录、重置密码、绑定/换绑等场景同理）。

## 5. 和「验证码已生效」的关系

注意顺序：

1. **先**把验证码写入 Redis（此时校验侧已可用）
2. **再**把发送任务入队

因此：

- 接口返回成功，含义是「验证码已签发，发送任务已入队」
- 邮件/短信因供应商故障稍晚才到，或队列自动重试后才到，都属预期
- 用户仍可在 TTL 内输入验证码完成登录/注册

这也是「验证码与发送解耦」的核心：校验依赖 Redis 里的码，不依赖第三方是否已成功投递。

---

## 6. 实现时注意点（本仓库）

1. **BullMQ 独立 Redis 连接**  
   现有 `RedisService` 使用了 `maxRetriesPerRequest: 1`，而 BullMQ 要求 `maxRetriesPerRequest: null`，因此队列单独建连接，不复用业务 Redis 客户端。

2. **失败记录要裁剪**  
   `removeOnComplete` / `removeOnFail` 限制 Redis 里堆积的 Job 记录，避免长期占满。

3. **防刷不靠队列**  
   队列解决的是「怎么发」；「能不能发」靠图形验证码 + 发送间隔锁 + 单日上限（`VerificationService` / `CaptchaService`）。

---

## 7. 什么时候可以不同步入队？

本地调试、极简单 demo 可以直接同步发。  
但只要邮件/短信走真实第三方、需要重试或会有并发发码，**异步队列就是更稳妥的默认选择**。本项目同时用它实践 BullMQ（见认证增强计划中的方案决策）。
