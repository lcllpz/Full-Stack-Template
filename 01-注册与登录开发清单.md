# 注册 + 邮箱密码登录：后端开发清单

> 面向「前端转后端」的学习文档。以本项目（NestJS 六边形架构脚手架）的注册与邮箱密码登录为参照，按**专业后端开发流程**梳理需要做哪些事、为什么这样做。
>
> 每一节都带 `- [ ]` 清单，可以边学边勾。建议按顺序推进。

---

## 0. 先建立心智模型（前端 → 后端的思维转换）

前端关注「页面怎么渲染、交互怎么响应」；后端关注「数据怎么存、请求怎么验证、状态怎么流转、出错怎么处理、谁能访问」。

一次登录请求在后端会经历这样一条链路（本项目的真实分层）：

```
HTTP 请求
  → Controller（路由层：收参数、定路由、定响应码）
  → DTO + ValidationPipe（入参校验：邮箱格式、密码长度…）
  → Service（业务逻辑层：查用户、校验密码、签发 token）
  → Repository 端口（数据访问抽象）
  → Repository 适配器（TypeORM 实现，真正读写数据库）
  → Mapper（数据库实体 ↔ 领域实体互转）
  → 数据库（PostgreSQL）
返回 LoginResponseDto（token / refreshToken / user）
```

**关键约定（六边形架构）**：业务逻辑（Service）只依赖「端口」抽象，不直接碰数据库。换数据库只需换适配器，业务代码不动。这是本项目最核心的设计思想。

- [ ] 理解上面这条「请求链路」
- [ ] 理解「领域实体（domain/user.ts）≠ 数据库实体（entities/\*.entity.ts）」，二者靠 mapper 转换
- [ ] 理解「Service 依赖端口、不依赖具体数据库实现」

## 1. 需求分析与接口设计（动手写代码之前）

专业后端的第一步不是写代码，而是**定义接口契约**：有哪些接口、收什么、返回什么、出错返回什么。

本项目注册/登录相关的接口（见 `src/auth/auth.controller.ts`）：

| 方法 | 路径                          | 作用                 | 成功响应码     |
| ---- | ----------------------------- | -------------------- | -------------- |
| POST | `/api/v1/auth/email/register` | 邮箱注册             | 204 No Content |
| POST | `/api/v1/auth/email/login`    | 邮箱密码登录         | 200 + 令牌     |
| POST | `/api/v1/auth/email/confirm`  | 邮箱确认（激活账号） | 204            |
| GET  | `/api/v1/auth/me`             | 查询当前登录用户     | 200            |
| POST | `/api/v1/auth/refresh`        | 刷新令牌             | 200            |
| POST | `/api/v1/auth/logout`         | 登出                 | 204            |

- [ ] 列出你要实现的接口清单（先聚焦：register + login）
- [ ] 为每个接口定义「请求体字段」和「响应体字段」
- [ ] 约定错误返回格式（本项目用 `422 UnprocessableEntity` + `{ errors: { 字段: '错误码' } }`）
- [ ] 理解为什么注册返回 `204`（注册后需先去邮箱确认，没有立即登录态）

## 2. 数据建模（用户表 / 领域实体）

登录注册的核心数据是「用户」。需要想清楚：用户有哪些字段、哪些唯一、哪些敏感。

参考 `src/users/domain/user.ts` 的核心字段：

- `id`、`email`（唯一）、`password`（加密存储、序列化时排除）、`provider`（email / google…）
- `firstName` / `lastName`、`role`（角色）、`status`（active / inactive）、时间戳

- [ ] 设计用户领域实体字段（`domain/user.ts`）
- [ ] 标记敏感字段：`password` 用 `@Exclude({ toPlainOnly: true })` 防止返回给前端
- [ ] 标记分组字段：`email` 等用 `@Expose({ groups: ['me'] })` 控制谁能看
- [ ] 设计数据库实体（`infrastructure/persistence/relational/entities/user.entity.ts`）
- [ ] 写 mapper：domain ↔ entity 互转
- [ ] 理解「status=inactive 表示注册后未激活」的状态机设计

> 注意：本项目规定**不要手写实体文件**，要用 CLI 生成器（见 `.claude/skills/generate/SKILL.md` 与 `docs/cli.md`）。学习时可以先手写理解原理，实战时用生成器保持迁移/DTO 同步。

## 3. 输入校验（DTO + class-validator）

后端**永远不信任前端传来的数据**，所有入参必须校验。这是前端转后端最需要补的安全意识。

参考 `src/auth/dto/auth-register-login.dto.ts` 与 `auth-email-login.dto.ts`：

```typescript
export class AuthRegisterLoginDto {
  @Transform(lowerCaseTransformer) // 邮箱统一转小写
  @IsEmail()
  email: string;

  @MinLength(6) // 密码至少 6 位
  password: string;

  @IsNotEmpty()
  firstName: string;
  // ...
}
```

- [ ] 为 register 写 DTO：邮箱格式、密码长度、必填项
- [ ] 为 login 写 DTO：邮箱 + 密码
- [ ] 邮箱用 `@Transform(lowerCaseTransformer)` 归一化（避免大小写导致重复账号）
- [ ] 理解全局 `ValidationPipe`（`src/main.ts`）如何自动触发校验
- [ ] 理解校验失败时框架自动返回 422，无需手写

## 4. 密码安全（加密存储）

**绝不能明文存密码**。本项目用 `bcryptjs` 加盐哈希。

参考 `src/users/users.service.ts` 的 `create()`：

```typescript
const salt = await bcrypt.genSalt();
password = await bcrypt.hash(createUserDto.password, salt);
```

登录时用 `bcrypt.compare(明文, 哈希)` 比对（见 `auth.service.ts` 的 `validateLogin`）。

- [ ] 注册时哈希密码后再存库
- [ ] 登录时用 `bcrypt.compare` 校验，**不要**自己解密对比
- [ ] 理解「盐（salt）」的作用：让相同密码产生不同哈希
- [ ] 校验失败时返回模糊错误（避免泄露「是邮箱错还是密码错」——本项目对密码错误统一返回 `incorrectPassword`）

## 5. 注册流程实现（register）

参考 `AuthService.register()`，完整流程：

1. 调 `usersService.create()` 创建用户（内部会查邮箱是否已存在 → 已存在抛 422）
2. 创建时设 `status = inactive`（未激活）、`role = user`
3. 用 JWT 签一个「邮箱确认 token」（带 `confirmEmailUserId`）
4. 调 `mailService.userSignUp()` 发确认邮件，邮件里带这个 token

- [ ] 实现 `register()`：创建用户 + 设置初始状态/角色
- [ ] 处理「邮箱已注册」的冲突（返回 422 `emailAlreadyExists`）
- [ ] 生成邮箱确认 token（JWT，独立 secret 与过期时间，见 `auth.config.ts`）
- [ ] 发送确认邮件（`mail` 模块；学习阶段可先打印 token 到日志，不接真实邮件服务）
- [ ] 实现 `confirmEmail()`：校验 token → 把用户 status 改为 active

## 6. 登录流程实现（login）

参考 `AuthService.validateLogin()`，完整流程：

1. 按邮箱查用户，查不到 → 422 `notFound`
2. 校验 `provider === email`（社交登录账号不能用密码登录）
3. 校验密码（`bcrypt.compare`）
4. 生成随机 `hash`，创建一条 **Session** 记录（支撑刷新/登出）
5. 签发 `token`（access）+ `refreshToken`，计算 `tokenExpires`
6. 返回 `LoginResponseDto { token, refreshToken, tokenExpires, user }`

- [ ] 实现 `validateLogin()`：查用户 → 校验 provider → 校验密码
- [ ] 生成 JWT（payload 含 `id`、`role`、`sessionId`）
- [ ] 同时生成 refreshToken（payload 含 `sessionId`、`hash`）
- [ ] 创建 Session 记录（见 `src/session`，为登出/刷新的「可撤销」能力打基础）
- [ ] 返回登录响应 DTO，确认 `password` 字段没有被序列化出去

## 7. JWT 鉴权与受保护路由

登录后，前端带着 token 访问受保护接口。后端用 Passport 的 JWT 策略验证。

参考 `src/auth/strategies/jwt.strategy.ts` 与 controller 上的 `@UseGuards(AuthGuard('jwt'))`。

- [ ] 配置 `JwtStrategy`：从 `Authorization: Bearer <token>` 提取并验签
- [ ] 用 `@UseGuards(AuthGuard('jwt'))` 保护 `/auth/me` 等接口
- [ ] 实现 `me()`：从 token payload 取 `id`，查并返回当前用户
- [ ] 理解 access token（短期）vs refresh token（长期）的分工
- [ ] 实现 `refresh()` 与 `logout()`（基于 session 的 hash 校验与失效）
- [ ] 理解为什么 JWT 策略里不查库（见 `docs/auth.md`），以及 session 如何补足「可撤销」

## 11. 推荐学习顺序（最小闭环优先）

按这个顺序做，能最快跑通一个「能注册能登录」的闭环：

1. 用户领域实体 + 数据库实体 + mapper（第 2 节）
2. 用户仓储端口 + 适配器（数据能存能查）
3. register DTO + `usersService.create`（含密码哈希、查重）（第 3、4、5 节）
4. login DTO + `validateLogin` + 签发 JWT（第 6 节）
5. JwtStrategy + `/me` 受保护接口（第 7 节）
6. 补 session、refresh、logout、邮箱确认（第 5、7 节）
7. 配置抽离、错误处理、测试（第 8、9、10 节）

## 13. 给前端开发者的概念速查

| 前端熟悉的                  | 后端对应概念                              |
| --------------------------- | ----------------------------------------- |
| 组件 props 类型校验         | DTO + class-validator（入参校验）         |
| 状态管理（store）           | 数据库 + Session（持久化状态）            |
| 路由（react-router）        | Controller 路由 + HTTP 方法/版本          |
| 拦截器（axios interceptor） | Guard / Interceptor / Pipe（NestJS 管道） |
| 环境变量（.env）            | ConfigModule 命名空间 + 类型安全读取      |
| localStorage 存 token       | 后端签发 JWT，前端存，后端验签            |
| 接口 mock                   | Swagger 接口契约 + e2e 测试               |
