# 后端落地功能文档（docs/back）

## 文档目的

作为**开源项目**的可读性资产：`docs/back` 下的文档不是 API 手册复读，而是**把 `apps/back` 里已落地的功能逻辑讲清楚**，让新贡献者、维护者和前端协作者能：

- 快速建立**整体直觉**（架构、请求链路、职责边界）
- 按**代码真实实现**理解「为什么这样设计」
- 通过**交叉引用**串联相关模块，避免重复维护

写作标准：**通俗易懂、逻辑严谨**——先整体后细节，概念有类比，结论有源码锚点。

## 何时使用本 Skill

用户显式要求时执行，例如：

- 「沉淀 xxx 功能文档」
- 「参考 dto 文档，写权限/审计的梳理文档」
- 「@docs/back 帮我把 xxx 模块逻辑整理成文档」

## 执行流程

```
任务进度：
- [ ] 1. 明确文档范围（单一功能域，避免一篇涵盖全后端）
- [ ] 2. 阅读源码（Controller → Service → Guard/Interceptor → Entity/Config）
- [ ] 3. 查阅 docs/back 已有文档，确定目录与交叉链接
- [ ] 4. 按模板起草（见 reference.md）
- [ ] 5. 自检：链路完整、层级边界清晰、源码路径可点击
```

### Step 1：划定范围

- 一篇文档对应**一个可独立理解的功能域**（如「DTO 与管道验证」「权限管理 RBAC」「邮箱密码登录」）。
- 子主题可拆为同目录多篇（如登录主文档 + 密码加密 + Session 方案）。
- 路径规范：`docs/back/<功能目录名>/<文档名>.md`（中文目录名可接受，与现有文档保持一致）。

### Step 2：读代码（必须）

优先梳理：

| 关注点 | 典型路径 |
|--------|----------|
| 入口 | `*.controller.ts` |
| 业务 | `*.service.ts` |
| 全局机制 | `main.ts`、`app.module.ts` |
| 守卫/拦截器 | `**/guards/`、`**/interceptors/`、`**/*.guard.ts` |
| DTO/校验 | `**/dto/*.dto.ts`、`utils/validate/` |
| 配置 | `config/**/config.ts` |
| 实体关系 | `**/entities/*.entity.ts` |

记录：**模块职责表**、**请求顺序**、**与其它模块的边界**（如 401 vs 403、管道校验 vs Service 业务校验）。

### Step 3：对齐已有文档

撰写前阅读 [reference.md](reference.md) 中的「现有文档索引」，避免：

- 重复大段内容（用「延伸阅读」链接代替）
- 与已有文档的守卫顺序、错误码描述矛盾
- 新建目录与已有主题重复

### Step 4：写作

使用 [reference.md](reference.md) 中的**文档模板**与**章节排序原则**。

硬性要求：

1. **开篇一段**：说明本文梳理什么、覆盖 `apps/back` 哪块、读者读完能回答什么问题。
2. **整体架构优先**：表格（层次 / 模块 / 职责）+ mermaid 流程图 + 「设计要点」 bullet。
3. **请求链路**：Guard → Pipe → Controller → Service，标注 HTTP 状态码语义。
4. **源码锚点**：关键结论引用真实路径，使用 `` `startLine:endLine:filepath` `` 代码引用格式。
5. **边界讲清**：易混淆概念用对比表（如入参 ValidationPipe vs 出参 ClassSerializerInterceptor）。
6. **交叉引用**：`> 延伸阅读` 块 + 文内 `[标题](../路径.md#锚点)` 链接。
7. **参考文档**：文末列出 NestJS / 第三方官方链接，不替代项目内梳理。
8. **语言**：正文使用中文；技术名词、类名、文件名保持英文原文。

### Step 5：自检清单

- [ ] 新人能否只看文档理解「请求从哪进、在哪被拦、失败返回什么」？
- [ ] 每个「设计要点」是否能在源码中找到对应？
- [ ] 是否与 `docs/back` 其它文档的术语、顺序一致？
- [ ] mermaid / 表格是否比大段 prose 更清晰？（优先用图和表）
- [ ] 是否删除了与代码不符的臆测描述？

## 章节排序原则（推荐）

按阅读认知顺序组织，而非按文件目录顺序：

1. 直觉 / 目的 / 核心原则
2. 全局请求链路（mermaid）
3. 技术栈或核心机制（配置、管道、守卫等）
4. 项目内组织方式（DTO 命名、目录、常量位置等）
5. 常见模式与实例（附项目真实代码）
6. 进阶话题（条件校验、缓存、Transform 等）
7. 错误响应 / 与相邻模块的关系
8. 参考文档

子文档（方案总结类）可简化为：**结论 → 流程 → 源码片段 → 与主文档关系 → 参考**。

## 不要做的事

- 不要写纯装饰性文档（无源码、无链路、无边界）
- 不要把 plan 草稿原样粘贴（plan 在 `docs/plan/`，back 文档应对齐**已落地代码**）
- 不要在一篇里塞多个 unrelated 功能
- 不要删除用户已有注释式说明；优化时保留有效信息
- 不要未经用户要求提交 git commit
